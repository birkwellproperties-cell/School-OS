# Tavaro Enterprise Platform Architecture

**Platform:** Tavaro Enterprise Platform  
**Abbreviation:** TEP  
**Company:** Tavaro Group LLC  
**Version:** 1.0  
**Status:** Governing platform architecture

---

# 1. Purpose

Tavaro Enterprise Platform is the shared foundation for Tavaro business systems.

Initial product consumers include:

```text
SchoolOS
FarmOS
ContractorOS
```

Future product consumers may include:

```text
ClinicOS
WarehouseOS
HospitalityOS
ChurchOS
ManufacturingOS
```

TEP provides reusable enterprise capabilities so each product can focus on its industry-specific workflows.

# 3. Shared Platform Principles

TEP must be:

- Multi-tenant
- Product-neutral where practical
- Permission-driven
- Workflow-enabled
- Auditable
- Mobile-ready
- Secure by default
- Observable
- Versioned
- Backward-compatible where possible
- Testable independently
- Usable by multiple Tavaro products

Shared platform code must not contain SchoolOS-specific terminology unless implemented through product configuration.

---

# 4. Identity Platform

The Identity Platform owns:

```text
profiles
organizations
organization_memberships
schools or workspaces as tenant units
invitations
session-aware user context
account lifecycle
localization preferences
```

Responsibilities:

- Connect Supabase Auth identities to application profiles
- Resolve active organization and product context
- Support users belonging to multiple organizations
- Support invitation and onboarding flows
- Manage suspended and archived access
- Expose authenticated actor information to all products

The Identity Platform does not own employee, student, contractor, patient, or customer business records.

---

# 5. Tenancy Platform

The Tenancy Platform owns:

- Organization hierarchy
- Workspace or product tenant context
- School, farm, company, or clinic tenant registration patterns
- Membership resolution
- Tenant switching
- Tenant-safe service boundaries
- Tenant-aware audit and reporting context

Each product may extend tenancy with industry-specific structures.

Examples:

```text
SchoolOS
Organization → School → Campus

FarmOS
Organization → Farm → Site

ContractorOS
Organization → Company → Project
```

# 21. Shared Package Strategy

The initial SchoolOS repository may keep TEP code inside:

```text
src/platform
src/design-system
src/services
src/shared
```

A separate package workspace should be introduced only when:

- Shared APIs are stable
- At least two products actively consume the code
- Versioning requirements are understood
- Independent package testing is available

Premature package extraction should be avoided.

The preferred sequence is:

```text
Build shared boundaries
→ Stabilize interfaces
→ Reuse in second product
→ Extract packages
```

# 22. Dependency Direction

Allowed dependency direction:

```text
Product Modules
      ↓
TEP Platform Services
      ↓
External Providers and Infrastructure
```

TEP must not import SchoolOS business modules.

SchoolOS modules may import TEP platform services.

Business modules should not directly import another module's repositories unless an explicit shared service is approved.

# 27. Architecture Rule

A capability belongs in TEP when:

- It is product-neutral
- Multiple modules need it
- Multiple Tavaro products are likely to need it
- Its behavior should be standardized
- It benefits from centralized testing and security

A capability belongs in SchoolOS when:

- It represents school-specific business meaning
- It uses education-specific terminology
- It applies only to school workflows
- It owns education records or rules
