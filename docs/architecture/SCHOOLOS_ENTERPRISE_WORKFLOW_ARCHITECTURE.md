# SchoolOS Enterprise Workflow Architecture

## Version 1 Architecture Baseline

## 1. Purpose

The SchoolOS Enterprise Workflow Architecture defines how business processes are:

* Designed
* Versioned
* Initiated
* Assigned
* Approved
* Escalated
* Audited
* Completed
* Cancelled
* Integrated with tasks, notifications, permissions, documents, reporting, and events

Workflow is a shared capability of the Tavaro Enterprise Platform.

SchoolOS modules must not implement isolated approval logic, disconnected status systems, or module-specific workflow engines.

Every governed business process must use the TEP Workflow Platform.

---

# 2. Workflow Architecture Principles

The workflow architecture is governed by the following principles:

1. Workflow definitions are separate from workflow instances.
2. Published workflow versions are immutable.
3. Every transition is permission-controlled.
4. Every action is auditable.
5. Workflow state changes must be transactional.
6. Tasks and notifications are generated by workflow events.
7. Approval authority may depend on role, amount, institution, campus, department, or risk.
8. Workflows must support delegation, reassignment, escalation, and expiry.
9. Mobile clients may participate in workflows but cannot bypass enterprise controls.
10. Completed or cancelled workflows remain historically accessible.
11. Workflow logic must be configurable without duplicating application code.
12. Sensitive workflows must support separation of duties.
13. Workflow definitions must be reusable across Tavaro products where the business pattern is shared.
14. Business entities remain the source of operational data; workflow instances govern process state and history.

# 4. Workflow Core Concepts

## 4.1 Workflow Definition

A workflow definition is a named business process template.

Examples:

```text
schoolos.procurement.request
schoolos.admissions.application
schoolos.finance.fee_adjustment
schoolos.academics.result_publication
schoolos.hr.leave_request
schoolos.inventory.stock_adjustment
```

A workflow definition contains:

* Workflow key
* Name
* Description
* Owning product
* Owning module
* Entity type
* Trigger type
* Current published version
* Configuration schema
* Activation status

## 4.2 Workflow Definition Version

Every published workflow is versioned.

Example:

```text
schoolos.procurement.request
Version 1
Version 2
Version 3
```

Once a version has active instances, it must not be altered.

Changes require a new definition version.

Existing instances continue under the version with which they started unless a formally approved migration is executed.

## 4.3 Workflow Instance

A workflow instance represents one execution of a workflow.

Examples:

* Procurement Request PR-2026-000145
* Admission Application APP-2026-00312
* Leave Request LR-2026-00088

A workflow instance records:

* Tenant
* Institution
* Campus
* Workflow definition and version
* Source entity type
* Source entity ID
* Current state
* Current step
* Priority
* Initiator
* Started time
* Due time
* Completion time
* Cancellation time
* Correlation ID
* Context data

## 4.4 Workflow Step

A step represents a unit of workflow responsibility.

Examples:

```text
Department Review
Finance Review
Procurement Review
Principal Approval
Goods Receipt Confirmation
```

A step may be:

* User task
* Approval
* Review
* Automated action
* Conditional gateway
* Parallel gateway
* Wait state
* Timer
* Integration action
* Sub-workflow
* Completion step

## 4.5 Transition

A transition moves a workflow from one state or step to another.

Examples:

```text
submit
approve
reject
request_changes
resubmit
cancel
escalate
delegate
complete
```

Every transition must define:

* Source state
* Target state
* Required permission
* Eligible actor
* Validation rules
* Required fields
* Required comment status
* Required attachment status
* Side effects
* Emitted events

## 10.2 Approval Limits

Approval limits must be configurable.

Example:

```text
Department Head: up to KES 50,000
Finance Manager: up to KES 250,000
Principal: up to KES 1,000,000
Board Approval: above KES 1,000,000
```

The workflow engine must evaluate limits using normalized amounts where currency conversion is required.

## 10.3 Separation of Duties

The platform must support rules such as:

* The requester cannot approve their own request.
* The person creating a supplier cannot approve the first payment to that supplier.
* The person entering results cannot be the final result publisher.
* The person preparing payroll cannot be the sole payroll approver.
* The person recording goods receipt cannot approve the related supplier invoice where policy prohibits it.
* The person requesting an inventory adjustment cannot complete the stock count used to justify it.

Separation-of-duties violations must block transitions and generate audit events.

# 18. Workflow Transaction Standard

A business transition must execute atomically where possible.

A successful transition should:

1. Validate the actor.
2. Validate permissions.
3. Validate workflow state.
4. Validate business rules.
5. Validate required fields and documents.
6. Record the workflow action.
7. Complete the current step.
8. Activate the next step.
9. Update the source entity state.
10. Generate tasks.
11. Write notification events.
12. Write audit events.
13. Write business events to the event outbox.
14. Commit the transaction.

If any required operation fails, the transition must roll back.

External communications and integrations should be processed asynchronously from committed outbox events.

# 26. Procurement Separation of Duties

At minimum:

* Requester cannot provide final approval.
* Supplier creator cannot independently approve supplier activation.
* Quote evaluator cannot alter supplier quote submissions.
* Purchase-order approver cannot approve beyond their limit.
* Receiving actor cannot modify the approved purchase-order price.
* Supplier invoice approver cannot bypass the three-way match.
* Payment approver must be separate from the payment preparer where staffing permits.
* Emergency exceptions require retrospective review.

# 38. Architecture Baseline Decision

The SchoolOS Version 1 workflow architecture is approved on the following basis:

* TEP owns the shared workflow engine.
* Workflow definitions are versioned and immutable after publication.
* Workflow transitions are service-controlled.
* Permissions and assignment rules are evaluated server-side.
* Workflow actions are append-only and auditable.
* Tasks and notifications are derived from workflow activity.
* Business events are written through the transactional outbox.
* Separation of duties and approval limits are first-class controls.
* Desktop, tablet, Android, and iOS participate in the same workflow model.
* SchoolOS modules may define business workflows but may not create independent workflow runtimes.
* Procurement, admissions, academic results, finance adjustments, HR leave, inventory adjustments, transfers, and welfare cases are governed processes from Version 1.

This document is now part of the SchoolOS Version 1 Architecture Baseline.
